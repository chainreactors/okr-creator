# OKR Creator — 改进模式库

OKR Creator 在生成 SKILL.md 时，基于六维诊断分数自动选择 2-4 个适用模式嵌入到输出文件中。

## 选择规则

1. 扫描六维诊断分数，匹配触发条件
2. 按优先级选择（P0 相关维度优先）
3. 最多嵌入 4 个模式（避免信息过载）
4. 将模式的"第一步"具体化为目标项目的实际文件/命令

---

## 模式清单

### 模式 1: 测试冷启动

**触发条件：** 交付质量 ≤ 2，测试覆盖率为 0 或接近 0

**策略：**
不要试图一步到位达到 80% 覆盖率。从最关键的 3 个模块/函数开始，每次提交代码时要求至少为变更的代码添加测试。

**执行步骤：**
1. 识别项目中最核心的 3 个文件/模块
2. 为每个核心模块编写 1 个 happy path 测试
3. 在 CI 中添加测试运行步骤（即使只有 3 个测试）
4. 之后每个 PR 要求至少为变更代码附带测试

**预期时间线：** 1-2 周建立基础，之后持续积累

---

### 模式 2: 文档冲刺

**触发条件：** 文档完善 ≤ 2，缺少 CONTRIBUTING.md / CHANGELOG / 架构文档

**策略：**
集中 1 周时间补齐核心文档。不追求完美，先有再好。

**执行步骤：**
1. Day 1-2: 创建 CONTRIBUTING.md（开发流程、PR 规范、代码风格）
2. Day 3: 创建 CHANGELOG.md（从 git log 提取历史版本变更）
3. Day 4-5: 补充 README 中缺失的章节（安装、使用、配置）
4. 可选：添加架构决策记录（ADR）

**预期时间线：** 1 周

---

### 模式 3: CI/CD 从零搭建

**触发条件：** 自动化 ≤ 2，无 CI/CD 配置

**策略：**
从最简单的 lint CI 开始，每周叠加一层。不要一次性搭建完整流水线。

**执行步骤：**
1. Week 1: 添加 lint / format 检查 CI（最低门槛）
2. Week 2: 添加测试运行 CI
3. Week 3: 添加构建/打包 CI
4. Week 4: 添加发布自动化（tag → release）

**预期时间线：** 4 周递进

---

### 模式 4: 巨石文件拆分

**触发条件：** 结构架构 ≤ 2，存在 500+ 行的单文件

**策略：**
识别最大的文件，按职责边界拆分为独立模块。每次只拆一个文件，确保不引入回归。

**执行步骤：**
1. 识别项目中最大的 3 个文件
2. 为最大文件绘制职责地图（哪些功能混在一起）
3. 按职责边界提取为独立文件，保留原文件作为入口（re-export）
4. 验证拆分后功能不变（运行测试或手动验证）

**预期时间线：** 每个文件 2-3 天

---

### 模式 5: Roadmap 启动

**触发条件：** 项目愿景 ≤ 2，无 roadmap 或方向不明

**策略：**
定义 3 个里程碑（近期 / 中期 / 远期），每个里程碑有明确的交付物。

**执行步骤：**
1. 回顾项目初始目标和当前状态的差距
2. 定义 Milestone 1（1 个月内）：最小可用版本或最紧迫的改进
3. 定义 Milestone 2（1 个季度内）：核心功能完整
4. 定义 Milestone 3（半年内）：生态/社区/商业化就绪
5. 写入 README 或独立的 ROADMAP.md

**预期时间线：** 1 天规划，持续跟踪

---

### 模式 6: 债务清零冲刺

**触发条件：** 历史债务 ≤ 2，TODO/FIXME 数量 > 20 或过时依赖 > 10

**策略：**
集中 1 周时间清理技术债务。设立"债务清零日"，每天清理 5-10 个 TODO。

**执行步骤：**
1. `grep -rn 'TODO\|FIXME\|HACK\|DEPRECATED'` 统计全部债务
2. 按影响和复杂度分为 S/M/L
3. 先清理所有 S 级（通常 5 分钟内可解决）
4. M 级逐个处理，每个作为独立 commit
5. L 级转化为 Issue 或 KR 专项跟踪

**预期时间线：** S 级 1-2 天，M 级 1 周，L 级按 KR 跟踪

---

### 模式 7: 依赖升级安全网

**触发条件：** 历史债务 ≤ 3 且有过时依赖，或安全告警未处理

**策略：**
逐个升级依赖，每次升级后运行完整测试。不要批量升级。

**执行步骤：**
1. 列出所有过时依赖（npm outdated / pip list --outdated 等）
2. 按安全风险排序（有 CVE 的优先）
3. 逐个升级 + 测试，一个 commit 一个依赖
4. 配置 Dependabot / Renovate 自动化后续升级

**预期时间线：** 2-3 天初始清理 + 长期自动化

---

### 模式 8: 可观测性建设

**触发条件：** 自动化 ≤ 3 且项目已上线运行，缺少监控/日志/告警

**策略：**
从最基础的错误日志开始，逐步建立可观测性体系。

**执行步骤：**
1. 添加结构化日志（替代 console.log / print）
2. 配置错误追踪（Sentry 或同类）
3. 添加关键业务指标埋点
4. 配置基础告警（错误率阈值）

**预期时间线：** 2 周基础建设
